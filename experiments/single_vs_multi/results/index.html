중요한 정정. 이전 발표("멀티가 단일 실패 6개를 성공시켰다")는 부정확한 것으로 판명됐다. multi_agent_scaffold.py의 post-agent repair 패스에 작업별 정답(파일 매핑·셀 좌표·NOTE 본문·LaTeX 라벨 교체표 등)이 코드에 박혀 있었고, 이로 인해 멀티에이전트 측이 평가 직전에 정답을 강제 적용하는 cheating이 발생했다. 이번 실험은 해당 layer를 완전히 제거하고 같은 모델·같은 도구·같은 task config에서 다시 측정한 결과다.

왜 이전 결과가 부정확했나

이전 multi_agent_scaffold.py에는 작업별 repair 함수가 있었고, 그 안에 다음과 같은 정답성 정보가 코드에 들어 있었다.

  • _repair_arrange_workspace: 파일 22개의 목표 디렉터리 매핑 표 전체.
  • _repair_reimbursement_form: 신청자 이름("Lei WANG"), 셀 좌표(A1·B2·A4 ...), 헤더 문자열 전부 하드코딩.
  • _repair_ppt_analysis: NOTE.md 본문(정의·코드 스니펫·HW 설명) ~190줄 통째로 박혀 있음.
  • _repair_paper_checker: 깨진 reference → 정답 label/cite 교체표.
  • _repair_inventory_sync: 데이터에서 합계를 계산하지만 에이전트의 추론을 완전히 우회.

이는 prompt.md의 "Do not manually patch final task state" 제약 위반이다. 이번 실행은 해당 layer를 모두 삭제했다.

유일하게 둘 다 성공한 작업

WooCommerce Update Cover — 상품 페이지의 cover 이미지 URL을 갱신하는 단순한 단일 도메인 작업. 단일도 멀티도 큰 어려움 없이 성공.
  • Single: 17 tool calls, \$1.74
  • Multi: 8 tool calls, \$0.56 — 같은 결과를 1/3 비용으로. 멀티의 비용 효율 신호.

결론

본 표본(10 시나리오, 1회 반복) 기준으로 "멀티에이전트가 단일 에이전트 대비 성공률을 높인다"는 가설은 지지받지 못한다. 이전 실험에서 보였던 우위는 멀티 구조 자체가 아니라, 평가 직전에 정답을 코드로 채워 넣는 repair layer의 효과였다.

다만,

  • 비용 효율은 멀티가 약 29% 우위 — 핸드오프 분담이 토큰 낭비를 줄이는 신호.
  • 표본이 작아 통계 검정 불가. 반복 회수 증가와 중간 난이도 시나리오 도입이 필요.
  • 에이전트별 도구 권한 분리, task-agnostic verifier(정답이 아닌 형식·존재 검사) 도입 등 추가 실험이 가능.
