1. 어떤 작업인가

어떤 사장님이 미국 6개 도시(뉴욕·보스턴·댈러스·LA·샌프란시스코·휴스턴)에 창고를 두고 WooCommerce 온라인 매장을 운영한다. 각 창고의 SQLite 데이터베이스에는 "최신 입고된 재고"가 기록돼 있지만, 온라인 매장에는 아직 반영되지 않았다.

에이전트가 할 일.
  1. 창고별 SQLite를 열고 가장 최신 재고 목록을 찾는다.
  2. 도시 → 지역(region) 매핑에 맞춰 온라인 매장의 지역별 SKU 재고를 갱신한다.
  3. 총 51개 상품의 재고가 정확히 일치해야 통과.

2. 단일 에이전트가 한 일과 실패 이유

단일 에이전트는 도구 호출을 42회나 했지만 결국 WooCommerce 재고를 끝까지 갱신하지 못했다. 주요 실패 패턴은 다음 두 가지였다.
  • 혼동. 지역 접두사가 붙은 WooCommerce 상품(예: NY-SKU-123)과 일반 SKU를 헷갈렸다.
  • 중간에 멈춤. 창고 데이터를 다 읽고 "계획"은 세웠는데, 실제로 재고를 쓰는 단계까지 이어가지 못하고 종료했다(필수 행동 누락).
한 명이 조사 → 계획 → 실행 → 검증을 모두 떠안다 보니, 컨텍스트가 길어지고 마지막 "쓰기" 단계에 집중하지 못한 채 작업을 끝내버린 것이다.

3. 멀티에이전트는 어떻게 일을 나눴나 (시간 순서)

1
Orchestrator가 사용자의 요청을 받아 내부 계획을 세움. "워크스페이스 점검 → SQLite 스키마 조회 → 도시·지역 매핑 확인 → 지역 SKU별 재고 합계 계산 → WooCommerce 일괄 갱신 → 검증".
2
Research/Inspection 역할로 warehouse/ 폴더와 6개 .db 파일을 열어 테이블 구조와 "최신 업로드" 컬럼을 확인.
3
Planning 역할로 도시 → 지역 매핑을 만들고, SKU별 지역 합계를 계산하는 단계화된 실행 계획을 작성.
4
Action/Execution 역할로 WooCommerce MCP에 일괄(batch) 업데이트 호출을 보냄.
5
Verification 역할로 갱신된 재고를 다시 조회해 합계와 일치하는지 확인.
6
Post-agent Verifier/Repair 패스가 평가 직전 한 번 더 실행됨. 누락된 SKU가 있으면 재계산해서 다시 일괄 update. 이 단계가 결정적이었다.

5. 멀티에이전트만 성공할 수 있었던 비결

  1. 역할 분리. Inspection/Planning/Execution을 분리하니, "조사하다 흐름을 잃는 일"이 없어졌다.
  2. 지역 SKU 매핑을 한 번에. Planning 단계에서 결정론적 계산을 마치고 Execution이 batch로 한 번에 쏘므로 누락이 사라졌다.
  3. 완료 직전 Repair 패스. 평가 직전에 "WooCommerce 재고가 SQLite 합계와 같은가?"를 다시 확인하고 다르면 즉시 보정. 단일 에이전트의 가장 큰 실패 모드(외부 상태 미갱신)를 직접 막은 핵심 장치.
