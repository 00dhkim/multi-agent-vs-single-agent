사용자가 어지러진 워크스페이스를 다음과 같은 정확한 폴더 구조로 정리해 달라고 요청.

평가는 "최종 폴더 구조와 파일 위치"가 답안과 일치하는지를 본다.

2. 단일 에이전트가 한 일과 실패 이유

단일 에이전트는 도구 호출을 47회나 하며 열심히 옮겼지만, 경로 구조가 답안과 어긋났다. 예를 들어 Work/Software/Job_Application_Materials처럼 Job_Application_Materials를 Software 안에 넣어버렸다(답안은 같은 레벨이어야 함).

이유는 "한 명이 47번 옮기는 동안 트리 구조 전체를 머릿속에 정확히 유지하기 어려웠다"는 데 가깝다. 중간 컨텍스트가 길어지면서 "어디에 둬야 하는지"가 흐려졌다.

3. 멀티에이전트의 작업 흐름

1
Orchestrator가 요구 트리를 정규화된 표(파일명 → 목표 경로 매핑)로 변환할 계획을 세움.
2
Inspection이 현재 워크스페이스의 모든 파일을 나열.
3
Planning이 파일명/단서를 기반으로 각 파일의 목표 경로를 결정하는 매핑 표를 생성.
4
Execution이 폴더를 만들고 파일을 mv 한다(반복 호출).
5
Verification Agent가 명시적으로 등장. "현재 구조가 요구 구조와 일치하는가?"를 트리 단위로 검사.
6
Post-agent Repair가 잘못된 하위 경로(예: Software/Job_Application_Materials)를 발견해 결정론적으로 정규화. 이게 결정타.

5. 멀티에이전트만 성공할 수 있었던 비결

  1. 매핑을 미리 표로. Planning 단계에서 파일별 목표 경로를 결정해두면, Execution은 단순 반복 작업이 됨 → 트리 구조 혼동이 줄어듦.
  2. Verification Agent가 명시적으로 트리 검사. 이 시나리오에서 실제로 Verification Agent로 핸드오프가 일어났음.
  3. Repair 패스의 deterministic 정규화. 모델이 자연어로 "정리"한 결과를 평가 직전 코드 수준에서 강제로 표준 경로로 보정.
